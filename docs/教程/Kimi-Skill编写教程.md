# Kimi Skill 编写教程

> 本文介绍如何为 Kimi Code CLI 编写自定义 Skill，让 AI 助手掌握特定领域的知识和流程。
> 
> **参考文档**：[Kimi Skill 标准格式](../参考/Kimi-Skill-标准格式.md) - 官方规范原文

---

## 什么是 Skill

Skill（技能）是给 Kimi AI 的**专用教程文件**。它告诉 Kimi 在特定场景下该做什么、怎么做。

**举个例子**：
- 没有 Skill：每次你都要教 Kimi "版本发布要先检查清单，再提交代码，再 bump 版本..."
- 有了 Skill：你说"收尾版本"，Kimi 自动按预定流程执行

**本质**：把重复教 AI 的流程，写成文件，以后自动调用。

---

## 什么时候需要写 Skill

| 场景 | 是否需要 Skill |
|------|---------------|
| 偶尔问一次的问题 | ❌ 不需要，直接对话 |
| 重复 3+ 次的流程 | ✅ 需要，写成 Skill |
| 多人协作的规范 | ✅ 需要，统一标准 |
| 复杂的领域知识 | ✅ 需要，AI 不具备 |

**本项目示例**：
- `release-assistant` - 版本发布流程（重复操作）
- `kimi-cli-help` - Kimi CLI 使用帮助（领域知识）

## Skill 文件结构

```
.kimi/skills/
└── skill-name/           # Skill 目录名
    ├── SKILL.md          # 必需：核心定义文件
    ├── scripts/          # 可选：可执行脚本
    ├── references/       # 可选：参考文档
    └── assets/           # 可选：模板资源
```

### SKILL.md 结构

```yaml

---
name: skill-name                 # Skill 标识符
description: 详细描述 + 触发条件  # 这行决定何时触发！
---

# 标题

## 触发条件
什么时候用这个 Skill

## 执行流程
步骤 1...
步骤 2...

## 可选资源
- 脚本：`scripts/xxx.py`
- 参考：`references/xxx.md`
```

## 核心原理：三级加载系统

Skill 采用**渐进式加载**设计，节省上下文窗口：

```
┌─────────────────────────────────────────┐
│  第一层：Metadata                       │  ← 始终在内存
│  (name + description)                    │     ~100 词
│                                           │
│  作用：匹配用户输入，决定是否触发       │
└─────────────────────────────────────────┘
                    ↓ 匹配成功
┌─────────────────────────────────────────┐
│  第二层：SKILL.md Body                  │  ← 触发后加载
│  (Markdown 正文)                         │     <5k 词
│                                            │
│  作用：执行具体流程                       │
└─────────────────────────────────────────┘
                    ↓ 需要时
┌─────────────────────────────────────────┐
│  第三层：Bundled Resources              │  ← 按需加载
│  (scripts/ references/ assets/)             │     无限制
│                                           │
│  作用：提供详细参考或执行脚本            │
└─────────────────────────────────────────┘
```

**类比**：
- 第一层 = 书的封面（决定是否买）
- 第二层 = 目录和简介（决定怎么看）
- 第三层 = 具体章节（需要时才翻）

## 编写步骤

### 步骤 1：确定触发条件（最重要）

`description` 字段决定 Skill 何时触发，必须写清楚：

**❌ 错误示例**：
```yaml
description: 版本发布助手
# 太模糊，Kimi 不知道什么时候该用
```

**✅ 正确示例**：
```yaml
description: SandTank 项目版本收尾和发布流程助手。当用户提及"收尾版本"、"推进版本"、"准备发版"、"结束x.x.x"、"bumpBuild"、"bumpPatch"等关键词时触发，提供版本发布的检查清单和流程指导。
# 明确：做什么 + 什么时候用
```

### 步骤 2：编写 SKILL.md 正文

**原则**：
1. **简洁** - 默认 Kimi 很聪明，只写它不知道的
2. **结构化** - 用标题、列表、表格组织
3. **示例优先** - 给例子，少解释

**模板**：

```markdown

---
name: my-skill
description: 做什么 + 触发词："xxx"、"yyy"
---

# Skill 标题

## 触发条件
- 关键词 1：说明
- 关键词 2：说明

## 核心流程

### 情况 A
1. 步骤 1
2. 步骤 2

### 情况 B
1. 步骤 1
2. 步骤 2

## 关键信息
| 选项 | 说明 |
|------|------|
| A | 说明 |
| B | 说明 |

## 可选资源
- 脚本：`scripts/tool.py` - 用途说明
- 参考：`references/guide.md` - 详细说明
```

### 步骤 3：添加可选资源（按需）

#### scripts/ - 可执行脚本

当需要精确执行或重复写代码时：

```python
# scripts/calculate.py
import sys

def main():
    # 计算逻辑
    pass

if __name__ == "__main__":
    main()
```

在 SKILL.md 中引用：
> 运行计算脚本：`python scripts/calculate.py`

#### references/ - 参考文档

内容太长时拆分：

```markdown
<!-- references/advanced.md -->
# 高级用法

详细内容...
```

在 SKILL.md 中引用：
> 详见 `[高级用法](references/advanced.md)`

**注意**：Kimi 只在需要时才加载这个文件。

#### assets/ - 模板资源

需要复制或修改的文件：

```
assets/
└── template.md
```

在 SKILL.md 中：
> 使用模板：`assets/template.md`

## 实战示例

### 示例：简单的代码审查 Skill

```yaml

---
name: code-review
description: 协助代码审查，检查常见问题和改进建议。当用户说"审查这段代码"、"review code"、"检查代码质量"时触发。
---

# 代码审查助手

## 触发条件
- "审查这段代码"
- "review code"
- "检查代码质量"

## 审查清单

### 1. 基础检查
- [ ] 代码是否能正常运行
- [ ] 是否有明显的语法错误
- [ ] 命名是否清晰

### 2. 质量检查
- [ ] 是否有重复代码
- [ ] 函数是否过长（>50 行）
- [ ] 是否有注释说明复杂逻辑

### 3. 安全/性能
- [ ] 是否有敏感信息硬编码
- [ ] 是否有明显的性能问题

## 输出格式

审查结果使用以下格式：

> ```markdown
> ## 审查结果
> 
> ### ✅ 优点
> - 优点 1
> - 优点 2
> 
> ### ⚠️ 建议改进
> - **位置**：文件名:行号
>   **问题**：描述
>   **建议**：具体改进方案
> 
> ### ❌ 必须修复
> - **位置**：文件名:行号
>   **问题**：描述
>   **风险**：说明
> ```
```

## 调试和迭代

### 测试 Skill

1. **保存文件** 到 `.kimi/skills/skill-name/SKILL.md`
2. **重启对话** 或等待 Kimi 重新加载
3. **触发测试**：使用 description 中的关键词

### 常见问题

| 问题 | 原因 | 解决 |
|------|------|------|
| Skill 不触发 | description 写得太模糊 | 明确关键词 |
| 触发太频繁 | 关键词太通用 | 加限定条件 |
| 内容太长 | 全塞进 SKILL.md | 拆到 references/ |
| 执行不对 | 步骤描述不清 | 给具体示例 |

### 迭代优化

使用 Skill 后发现不足：

1. **记录问题** - 哪一步 Kimi 没做对
2. **修改 SKILL.md** - 补充说明或示例
3. **测试验证** - 再次触发看效果
4. **精简内容** - 删除 Kimi 已懂的内容

---

## 总结

| 要素 | 要点 |
|------|------|
| 触发 | `description` 必须明确关键词 |
| 简洁 | 默认 Kimi 聪明，只写必要的 |
| 结构 | 标题、列表、表格组织内容 |
| 分层 | SKILL.md 精简，细节放 references/ |
| 示例 | 给具体例子，少抽象解释 |

**一句话**：Skill = 触发词 + 执行流程 + 按需加载的资源，越精简越好。
