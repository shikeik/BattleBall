# 代码质量与重构 (Code Quality & Refactoring)

## 核心原则：DRY (Don't Repeat Yourself)

> 不要重复你自己。任何知识在系统中都应有单一、明确、权威的表示。

---

## 五项重构指标

### 1. 提取方法/变量 (Extract Method/Variable)
- **症状**：重复代码块、魔法数字、过长函数
- **解决**：提取公共函数、命名常量、拆分职责
- **示例**：
  ```javascript
  // 重构前
  if (x > 100 && y < 50) { ... }  // 魔法数字
  
  // 重构后
  const MAX_SIZE = 100;
  const MIN_SIZE = 50;
  if (isValidSize(x, y)) { ... }
  ```

### 2. 消除重复代码 (DRY)
- **症状**：相同/相似代码出现在多处
- **解决**：提取公共函数、使用配置、继承/组合
- **检查**：修改一个逻辑需要改几处？

### 3. 单一职责原则 (SRP - Single Responsibility Principle)
- **症状**：一个函数/类做太多事
- **解决**：拆分函数、分离关注点
- **标准**：一个函数只做一件事，一个类只有一个变化理由

### 4. 开闭原则 (OCP - Open/Closed Principle)
- **症状**：加新功能要改很多旧代码
- **解决**：抽象接口、配置驱动、插件化设计
- **目标**：对扩展开放，对修改关闭

### 5. 配置与代码分离
- **症状**：业务逻辑和数值硬编码混在一起
- **解决**：CONFIG 对象、外部配置文件、数据库
- **收益**：策划改数值不用动代码

## 重构触发器：迭代难易度评估

### 评估方法
对每个代码部分，检查是否命中以下情况：

| 命中项数 | 难易度 | 行动 |
|---------|--------|------|
| 0-1 项 | 低 | 可接受，继续观察 |
| 2 项 | **中** | **建议重构** |
| 3+ 项 | **高** | **必须重构** |

### 评估示例

**场景：敌人生成逻辑**

```javascript
// 当前代码检查
function spawnEnemy() {
    const type = Math.random() > 0.5 ? 'big' : 'small';  // 魔法数字 0.5
    const x = [10, 20, 30][Math.floor(Math.random() * 3)];  // 硬编码位置
    const hp = type === 'big' ? 100 : 50;  // 逻辑重复
    const speed = type === 'big' ? 2 : 5;  // 逻辑重复
    // ... 创建敌人
}
```

**命中检查：**
- [x] 提取方法/变量 - 魔法数字 0.5，位置硬编码
- [x] 消除重复 - type判断重复了两次
- [ ] 单一职责 - 只做生成一件事，OK
- [ ] 开闭原则 - 加新类型要改这里
- [x] 配置分离 - 数值都硬编码

**结果：3 项命中 → 难度高 → 必须重构**

## 重构步骤

### Step 1: 识别代码异味 (Code Smell)
- 重复代码
- 过长函数
- 过大类
- 魔法数字
- 发散式变化

### Step 2: 应用重构手法

| 问题 | 手法 |
|------|------|
| 重复代码 | 提取函数、上移方法 |
| 过长函数 | 提取函数、替换算法 |
| 魔法数字 | 提取常量、配置化 |
| 数据泥团 | 提取类、引入参数对象 |
| 开关语句 | 多态、策略模式 |

### Step 3: 验证与测试
- 重构前后行为一致
- 单元测试通过
- 代码审查通过

## 最佳实践

### 小步快跑
- 每次只改一点
- 频繁提交
- 随时可回滚

### 先写测试再重构
- 确保重构不破坏功能
- 测试即文档

### 命名即文档
```javascript
// 差
function calc(a, b) { return a * b; }

// 好
function calculateRectangleArea(width, height) { 
    return width * height; 
}
```

### 配置层级
```javascript
// 硬编码 → 常量 → 配置对象 → 外部文件 → 远程配置
const CONFIG = {           // 小项目
    ENEMY: { hp: 100 }
};

// config/enemy.json       // 中项目
// 远程配置服务            // 大项目
```

## 常见反模式

### 过早优化
- 不要为了"可能"的需求过度设计
- 先让代码工作，再让代码好看，再让代码快

### 过度工程
- 简单问题复杂化
- 滥用设计模式

### 重构恐惧症
- "能跑就别动"
- 技术债务越积越多

## 快速检查清单

重构前问自己：
- [ ] 这段代码修改时需要改几处？
- [ ] 加新功能要改旧代码吗？
- [ ] 这个函数能做一件事吗？
- [ ] 这些数字/字符串有明确含义吗？
- [ ] 配置和逻辑混在一起了吗？

**2项以上勾选 → 考虑重构**
