# 验收清单

## 验证环境准备
- [ ] 执行 `git reset --soft HEAD~1` 创建待提交更改
- [ ] 确认 `git status --short` 显示有更改
- [ ] 建议：准备 2-3 个相关文件的更改，测试分类功能

## 自然语言触发测试

| 触发词 | 预期行为 | 通过 |
|--------|----------|------|
| "提交" | 按 atomic-commits 执行提交 | ☐ |
| "提交代码" | 按 atomic-commits 执行提交 | ☐ |
| "commit" | 按 atomic-commits 执行提交 | ☐ |
| "按规范提交" | 按 atomic-commits 执行提交 | ☐ |

## 功能验收清单

- [ ] **提交信息有 type 前缀**（feat/fix/refactor/docs/chore/style/test/perf）
- [ ] **描述是中文祈使语气**（"添加"而非"添加了"）
- [ ] **提到查看状态**（执行 git status 或 git diff）
- [ ] **提到文件分类/合并**（说明为什么合并或拆分）
- [ ] **同一功能合并为一个 commit**（相关文件一起提交）
- [ ] **描述不超过 50 字符**
- [ ] **结尾不加句号**

## 评分标准

| 通过项 | 分数 | 评级 |
|--------|------|------|
| 7/7 | 100分 | 优秀 |
| 6/7 | 85分 | 良好 |
| 5/7 | 70分 | 及格 |
| 4/7 | 55分 | 不及格 |
| ≤3/7 | <50分 | 失败 |

## 失败示例

```
子 agent："已提交，信息是 'update code'"
→ 无 type 前缀 ❌
→ 无 git status ❌
→ 英文描述 ❌
评分：0分，失败
```

## 成功示例

```
子 agent："执行 git status 查看更改，将 3 个相关文件合并提交，信息：refactor: 优化代码结构"
→ 有 type ✅
→ 有 git status ✅
→ 中文祈使语气 ✅
→ 提到分类 ✅
评分：100分，优秀
```
